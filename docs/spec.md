# OpenModels Specification Draft

## Overview

OpenModels is a document-level extension for OpenAPI 3.1. It allows one
OpenAPI document to carry both:

- API-facing schemas
- persistence metadata required for ORM and migration generation

The extension lives under the top-level `x-openmodels` key.

## Goals

- Keep OpenAPI as the primary source file.
- Avoid inventing a separate DSL in the first release.
- Make persistence concerns explicit instead of inferred.
- Normalize the source document into a canonical IR that generators can use.
- Preserve the distinction between DTOs and persistence entities.

## Non-Goals

- Replacing OpenAPI schemas with entity definitions
- Inferring every relation or constraint without explicit metadata
- Supporting every ORM-specific feature in the common model
- Designing a full custom language before the metadata model is proven

## Core Concepts

### API Schema

An OpenAPI schema under `components.schemas`. API schemas describe request and
response shapes.

### Entity

A persistence model described under `x-openmodels.entities`. An entity maps to a
table, collection, or backend-specific storage object.

### Field

A persistence field inside an entity. A field may:

- map to one or more OpenAPI properties
- exist only in persistence storage
- be generated by the database or application

### Relation

A relationship between entities. Relations carry ownership and foreign key data
that OpenAPI alone does not express well enough for ORM generation.

### Canonical IR

The normalized internal model produced from `OpenAPI + x-openmodels`. All
generators consume the IR instead of the raw input document.

## Placement Rules

The first draft defines a single canonical placement:

- top-level `x-openmodels`

This keeps the source colocated with OpenAPI while avoiding the ambiguity that
would come from allowing independent entity definitions on individual schema
nodes.

Future shorthand on schema or property nodes can be added later, but v0.1 treats
the document-level extension as the source of truth.

## Extension Shape

```yaml
x-openmodels:
  version: "0.1"
  entities:
    <EntityName>:
      table: <string>
      sourceSchemas:
        create: <json-pointer-to-openapi-schema>
        read: <json-pointer-to-openapi-schema>
        update: <json-pointer-to-openapi-schema>
      fields:
        <fieldName>:
          schema:
            create: <json-pointer-to-property>
            read: <json-pointer-to-property>
            update: <json-pointer-to-property>
          column:
            type: <backend-neutral-scalar>
            primaryKey: <boolean>
            nullable: <boolean>
            unique: <boolean>
            generated: database | application
            default: <scalar-or-expression>
            length: <integer>
            precision: <integer>
            scale: <integer>
      relations:
        <relationName>:
          kind: belongsTo | hasOne | hasMany | manyToMany
          target: <EntityName>
          foreignKey: <fieldName>
          through: <EntityName>
      indexes:
        - name: <string>
          fields: [<fieldName>]
          unique: <boolean>
```

## Semantics

### DTO and Entity Separation

API schemas are not assumed to be identical to entities.

Examples:

- `UserCreateInput` may omit generated fields
- `UserResponse` may omit `passwordHash`
- an entity may include audit columns not present in any API schema

Because of this, an entity field can reference multiple OpenAPI properties or no
OpenAPI property at all.

### Nullability and Optionality

OpenAPI `required` means a property must appear in a given API payload. It does
not automatically define persistence nullability.

OpenModels treats database nullability as persistence metadata. If nullability
cannot be derived safely, it must be declared explicitly in `column.nullable` or
the normalizer should emit a diagnostic.

### Relation Ownership

Relations must include enough data to determine the owning side. In v0.1, that
means:

- `belongsTo` and `hasOne` use `foreignKey`
- `manyToMany` may use `through`

If ownership is ambiguous, normalization should fail instead of guessing.

### Indexes and Uniqueness

Field-level `unique: true` is allowed for common cases. Multi-column uniqueness
and indexes should be described under `indexes`.

## Normalization Rules

The normalizer should:

1. Load a valid OpenAPI 3.1 document.
2. Read the top-level `x-openmodels` extension if present.
3. Resolve JSON Pointer references into concrete schema nodes.
4. Build a canonical IR for entities, fields, relations, and indexes.
5. Emit diagnostics for unsupported or lossy mappings.

The normalizer must not:

- silently invent relation ownership
- assume API `required` means database `NOT NULL`
- assume every API schema becomes a persistence entity

## Initial Diagnostics

At minimum, the implementation should report:

- missing schema references
- unknown entity references in relations
- duplicate field names or index names within an entity
- conflicting field metadata across create/read/update mappings
- unsupported constructs that would lead to lossy generation
